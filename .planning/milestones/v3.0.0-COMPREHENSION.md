## 1. What is this?

**At a high level:** v3.0.0 changes what the LocalGround Toolkit *is* — from "five markdown prompts you paste into Claude Code" to "three install paths the user picks based on how much friction they tolerate." The toolkit does the same job (safely migrate Claude Code projects off cloud-synced storage), but now anyone integrating with Claude Code can call its operations as native tool calls without copy-pasting a prompt every time.

| What shipped | Plain-language description |
|---|---|
| `@localground/mcp@3.0.0` | A small Node.js server that exposes 9 LocalGround operations as Claude Code tools. Once registered, Claude Code can call `localground_detect`, `localground_seed`, `localground_copy`, etc. directly — no prompt copy-paste, no per-session setup. |
| `@localground/cli@3.0.0` | A standalone command-line tool. Type `npx -y @localground/cli detect` and it runs without installation. Useful when you want one-off commands or scripts, not a Claude Code session. |
| `@localground/core` | An internal shared library both packages above use. Not published separately; bundled into the two installable packages. |
| 5 Claude Code skills | `/localground:seed`, `/localground:migrate`, `/localground:reap`, `/localground:cleanup`, `/localground:verify` — each one wraps one or more MCP tools with conversational guidance and confirmation prompts. |
| v2.0.0 prompts (preserved) | The five original markdown files still work, untouched, in `prompts/`. People who can't or won't install Node.js still have a working path. |
| Test suite + CI + release pipeline | 79 automated tests run on every push (Windows, macOS, Linux). New versions publish to npm via a token-less, signed automated workflow when a tag is pushed. |

**Scale:** 4 phases, 30 implementation plans, 14 days (April 13–26), 160 commits, ~5,877 lines of TypeScript. Two follow-up clean-up PRs (test fix for Windows-CI quirk; README + repo "About" refresh).

## 2. Why this approach?

**At a high level:** The v2.0.0 prompts grew to ~3,400 lines of natural-language instructions, and the most-recent independent evaluation found 46 issues — most of which traced to *ambiguity in deterministic operations*. Things like "decode this directory name into a filesystem path" or "verify these checksums match" are mechanical, but expressed as English instructions they got interpreted differently each run. The hybrid architecture (deterministic operations in code; user-judgment moments in skills/prompts) was chosen specifically to remove that ambiguity at its source.

| Alternative | Why it was rejected |
|---|---|
| Stay v2.0.0 only — keep iterating on prompt language | Independent evaluation showed the ambiguity wasn't a wording problem — it was a category problem. Re-wording deterministic operations in English makes them slightly clearer, never definitive. |
| Pure CLI / pure code | Loses the LLM-judgment moments that make skills feel guided rather than mechanical. Confirmation gates, natural-language reports, "what should I do next" interpretation — all weaker as raw CLI output. |
| Pure MCP server, no standalone CLI | Locks users into Claude Code as the only entry point. Some users want scripted automation or one-off terminal commands. Adding the CLI was a small marginal cost given the shared core library. |
| Squash all v3.0.0 work into one PR commit | Loses the per-plan commit history that documents *why* each design choice was made. The cherry-pick-with-filtering approach kept ~145 substantive commits while stripping the internal scaffolding. |
| Mocked filesystem tests | Phase 14 found Windows reparse-point bugs and OneDrive-path-decoding bugs that *only surface against a real filesystem*. Mocked tests would have shipped a broken decoder. |
| Push v3.0.0 git tag to GitHub | Pushing the tag triggers the new release pipeline, which would try to re-publish v3.0.0 (already manually published) and fail with a "version exists" error. Skipped intentionally; v3.0.1 will be the first tag-anchored, signed release. |
| Add v2.0.0 entry retroactively to MILESTONES.md | Out of scope for the v3.0.0 close; would have expanded scope without changing v3.0.0's archive accuracy. Captured as a known prior gap, not addressed. |

**The filter that rejected alternatives:** *removes ambiguity at the source vs. patches it later.* Anything that left the deterministic operations in natural-language form (option 1, partially option 2) was rejected. Anything that compromised the safety model (no candidate did) would have been rejected harder.

## 3. What would break?

**At a high level:** v3.0.0 has known fragilities. The core code is well-tested, but two automation pipelines have only been verified on paper, five end-to-end skill-routing tests were deferred, and the next release (v3.0.1) is the first time several of these contracts actually run live. The reviewer should know which risks were consciously accepted versus which are open-ended question marks.

**Failure modes still on the table:**

| Risk | Scenario → Mechanism → Consequence |
|---|---|
| First-tag-push release fails | The release pipeline uses a token-less trust contract between GitHub and npm (mechanism: OpenID Connect + provenance attestation, configured per-package on npmjs.com). If the trust contract was set up with a slightly wrong repository name, branch, or workflow filename, the very first publish attempt at v3.0.1 will fail with a `403 Forbidden` error and need adjustment before retry. (Captured as backlog 999.2.) |
| Five `/localground:*` skills are statically valid but never end-to-end tested | The skill files declare which MCP tools they pre-approve, the MCP tools exist, the underlying CLI commands work — but the full "user invokes `/localground:migrate` → MCP server runs the two-session migration with continuation tokens → handoff state file is written and read by Session 2" round-trip has never executed. Test 15 specifically (the two-session migrate skill) covers behavior nothing else exercises. (Captured as backlog 999.1.) |
| The local v3.0.0 git tag points at a commit *before* the milestone-close work | If a future contributor checks out `git tag v3.0.0` to reproduce what's on npm, they'll see the package source but not the milestone archive, ROADMAP reorganization, or PROJECT.md evolution. The tag accurately marks "what's on npm"; it does not mark "what's on master after milestone close." Two facts pointing at two different commits is a known mismatch. |
| Decoder is filesystem-dependent | The decoder reconstructs paths by listing actual filesystem entries and prefix-matching. If a folder is deleted between encode and decode, the decoder returns "no candidates" — silent in `audit` output unless the user explicitly compares pre/post counts. Phase 14 closed the worst manifestations; edge cases at folder boundaries (broken symlinks, permission-denied subtrees) could still cause silent drops. |
| `tsc --build` strict-mode is not a CI gate | The tests pass and the build artifacts work, but a separate type-checking pass reports ~30 strict-mode warnings that the bundler (mechanism: tsup uses esbuild, which is more permissive than tsc) tolerates. If a future change introduces a type error that tsup also tolerates, CI won't catch it. (Captured as backlog 999.3.) |

**Risks consciously accepted at close (not gaps):**

- **v3.0.0 lacks signed-publish provenance.** Manually published from a local terminal because npm's trusted-publisher feature requires the package to exist before the trust contract can be configured (npm/cli issue #8544). v3.0.1 onward will have provenance. Single-version trade-off.
- **Vitest emits a 10-second close-timeout warning at the end of the test run.** Cosmetic; tests pass correctly. (Captured as backlog 999.3.)
- **Published tarballs include `src/`, `test/`, and config files.** Inflates download size roughly 2x. Not a correctness issue. (Captured as backlog 999.4.)

**Review:** `[X] AFFIRMED · [ ] REWRITE · [ ] UNCERTAIN`

## 4. What did I learn?

**At a high level:** v3.0.0 produced six lessons that generalize beyond this milestone — patterns that apply to any future milestone, not just LocalGround. Three of them were already in memory before this work; three are new this session and have been captured as durable feedback memories. One is a moment where I (Claude) was confidently wrong and needed correction mid-execution.

**Generalizable patterns (already in memory or now added):**

1. **State files drift silently during long phase work.** REQUIREMENTS.md and ROADMAP.md showed Phase 12–13 as "in progress" at milestone close, despite the actual work having shipped weeks earlier. State updates have to happen *as part of phase close*, not at milestone close. (Already in memory: "GSD phase completion gaps".)
2. **Real-filesystem test fixtures catch platform bugs that mocked filesystems cannot.** Phase 14 specifically found Windows reparse-point handling and OneDrive-mixed-punctuation-path bugs that would have stayed invisible under mocked tests. The convention "use real `os.tmpdir()` + `mkdtemp` + `afterEach` cleanup, never `vi.mock('node:fs')`" is now a documented project rule.
3. **Windows CI uses DOS-era short paths even when local Windows doesn't.** GitHub's `windows-latest` runner returns paths like `C:\Users\RUNNER~1\...` from the standard temp-directory function, while `fs.readdir()` returns long names like `runneradmin`. Any test that round-trips a path through encoding has to canonicalize via `fs.realpath()` first. (New memory captured this session.)
4. **First-time npm publishing has a chicken-and-egg problem.** The token-less trust contract that's supposed to authenticate publishes can only be configured after the package exists on npm. The fix is to plan a one-time manual publish (mechanism: `npm login` + `npm publish` from a local terminal, no signed provenance) and configure the trust contract afterward, with all subsequent versions using the automated path. (Already in memory: "npm first publish requires manual flow".)
5. **When a bundler inlines a dependency, that dependency belongs in `devDependencies`, not `dependencies`.** Phase 15 caught this minutes before publish: the manifest declared `@localground/core` as a runtime requirement, which would have made every consumer install fail with "Cannot find package" because `@localground/core` was never published — only its source was bundled into the published packages. (Already in memory: "Workspace deps in devDependencies when bundled".)
6. **The `/gsd-pr-branch` skill's naive cherry-pick loop fails on transient-file conflicts.** When stripping `.planning/phases/*` from the staging area, subsequent commits that *modify* those stripped files can't apply because the prior file state has been removed from the branch. The fix (auto-resolve conflicts when *all* conflicted files are transient) worked here but should probably be upstreamed to the skill itself. Without the fix, 29 of 145 commits were dropped on first attempt.

**A moment I was confidently wrong:**

When you asked "do it" for the PR-branch creation, I ran the workflow as written without first checking whether the data shape (many phase-PLAN.md files modified across many commits) would conflict with the workflow's stripping logic. 29 commits failed on the first cherry-pick run. I had to revise the approach mid-execution. The lesson is for me, not for you: when a workflow has assumptions about data shape (here: that conflicts during cherry-pick won't happen, or will be on non-stripped files), pre-flight check the data before committing to the algorithm. The user trusted me to execute and I should have surfaced the risk before starting — not after the first 29 failures.
