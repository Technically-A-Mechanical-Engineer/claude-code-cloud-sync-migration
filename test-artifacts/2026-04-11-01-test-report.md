## Findings

### F-01: Robocopy requires MSYS_NO_PATHCONV=1 on bash-on-Windows [Migration, Medium]

**What happened:** Phase 4 robocopy calls failed twice before Claude discovered that `MSYS_NO_PATHCONV=1` is needed to prevent Git Bash from converting Windows-style paths. First attempt used Unix-style paths (robocopy doesn't understand them). Second attempt used `cmd.exe /c` wrapping (quoting broke). Third attempt with `MSYS_NO_PATHCONV=1` worked.

**Impact:** Every migration on bash-on-Windows will burn 2 failed attempts per folder before the first success, then Claude learns and uses it for subsequent folders. Wastes time and tokens on the first folder.

**Fix:** In the migration prompt's Phase 4 copy instructions, add `MSYS_NO_PATHCONV=1` to the bash-on-Windows robocopy command template. Something like:

```
**bash-on-Windows:**
```bash
MSYS_NO_PATHCONV=1 robocopy "C:\source\path" "C:\target\path" /E /COPY:DAT /DCOPY:T /R:1 /W:1
```

**Location in prompt:** Phase 4.2 copy commands, bash-on-Windows variant.

---

### F-02: Pre-flight step 2.4 wording implies closing the active session [Migration, Low]

**What happened:** Step 2.4 says "Close VS Code, any other Claude Code sessions, and any editors/terminals with open handles on the source folders." The phrase "any other Claude Code sessions" is correct but could confuse users into thinking they need to close the session that's running the migration.

**Impact:** Confusion risk only — a user who closes the active session loses the migration context. Low probability since most users will realize the session they're in is the one doing the work.

**Fix:** Add a clarification line:

> "Close VS Code, other Claude Code sessions, and any editors/terminals with open handles on the source folders. **This session stays open — it's running the migration.**"

**Location in prompt:** Phase 2.4.

---

### F-03: sed errors during path-hash decoding [Verification, Low]

**What happened:** Phase 3 path-hash decoding produced sed errors:
```
sed: -e expression #1, char 11: unterminated `s' command
sed: -e expression #1, char 7: unterminated `s' command
```

The prompt recovered — all 22 entries were decoded and classified correctly despite the errors. The errors are likely a bash-on-Windows quoting issue where special characters in the path-hash names (consecutive hyphens from spaces/commas) interact with sed's substitution syntax.

**Impact:** Cosmetic — errors appear in the output but don't affect results. User might lose confidence seeing error messages during an audit.

**Fix:** Investigate the sed command in the verification prompt's Phase 3 path-hash decoding logic. May need to use a different quoting strategy or switch to a different tool (awk, parameter expansion) for bash-on-Windows.

**Location in prompt:** Phase 3.1 or 3.2 (path-hash decode logic).

---

### F-04: Parallel tool calls cancelled under manual approval mode [Migration, Informational]

**What happened:** With "dangerously accept permissions" turned off, Phase 1.4 and 1.5 ran as parallel bash calls. One call errored (likely due to approval timing) and cancelled the other. Turning auto-accept back on resolved it. The prompt recovered and continued.

**Impact:** Users who step through permissions manually may see cancelled operations and confusing error messages. The prompt recovers, but the experience is rougher.

**Fix:** This is a Claude Code CLI behavior, not a prompt bug. Two options:
1. Add a note in the human-readable section recommending auto-accept for best results
2. Reduce parallel tool calls in early phases where sequential execution is acceptable (Phase 1 is low-cost — parallelizing 1.4 and 1.5 saves seconds, not minutes)

**Location:** Human-readable section (option 1) or Phase 1.4/1.5 instructions (option 2).
